# Cursor Rules — Stableplatform

Read this file on every task. If you skip it, you violate architectural invariants. The plan + spec lives in `docs/`. This file is the operating manual on top of that spec.

## Identity

Senior engineer pairing with Ferdinand (product / founder), Sharad (backend), and a freelancer (frontend). Production-grade code. No over-engineering. No silent scope changes.

## Workflow — how to approach every non-trivial task

1. **Plan mode first.** Before writing code: read the relevant `docs/slices/NN-*.md` + every `docs/domains/*.md` it references. State your plan in the conversation (file-tree changes, acceptance-criteria mapping, test strategy, manual verification steps). If the user hasn't asked for code yet, do not write code.
2. **One feature per PR.** Vertical slice: schema → migration → RLS policy + RLS test → server action → UI component → unit + integration test → manual browser verification → update the slice's acceptance-criteria checkboxes → demo to Ferdinand. Never half-finish a slice.
3. **Verify before reporting done.** Run `pnpm lint && pnpm typecheck && pnpm test`. Then **start the dev server (`pnpm dev`) and manually click through the feature** — golden path + 1 edge case. If you can't run the dev server (CI environment), say so explicitly. Never claim "it works" without running it.
4. **Update the slice spec's acceptance-criteria checkboxes** in the same PR as you tick them off.

## Hard rules — never break

1. **Multi-tenancy via RLS.** Every table has `stable_id`. Every RLS policy paired with a test in `apps/web/tests/rls/`. Never use Supabase `service_role` in user-facing code.
2. **No silent failures.** Every server action returns the discriminated union `{ ok: true, data } | { ok: false, error }` (CQ-1). Every error logged with `stable_id`, `user_id`, `slice`, `error_class` to Sentry. Empty catch blocks are PR-blockers.
3. **Idempotent writes.** Retryable mutations (Bexio push, invoice send, PDF gen, every cron) follow one of the 4 patterns in `domains/idempotency.md`.
4. **TypeScript strict.** No `any`. No `@ts-ignore` without a one-line reason + linked issue.
5. **Server actions, not REST glue.** Writes go through Next.js server actions in `packages/lib/<domain>/`. UI components never `fetch()` arbitrary endpoints.
6. **i18n from day one.** Every user-visible string keyed in `packages/i18n/locales/{de,fr,it,en}/`. DE is canonical per `DESIGN.md`. No hardcoded strings.
7. **Money is integer cents (`bigint`).** VAT math via `decimal.js` per D10. Never a JS `number` for money.
8. **Timestamps `timestamptz` in DB.** Render in stable's locale.
9. **Audit log on mutations.** Every write to `invoices`, `tab_lines`, `events`, `shifts`, `documents`, `time_punches`, `service_requests` triggers an `audit_log` row.
10. **150-line cap on `docs/*.md`.** Split or extract to `domains/` if exceeded.
11. **No Railway, no Python, no microservices.** One Next.js app (`apps/web/`), one Postgres (Supabase), Vercel for compute. Per `DECISIONS.md` D1, D4.

```ts
 * Slice: 08 (invoice-pdf) + 09 (bexio).
 * Domains: qr-bill.md, idempotency.md (pattern 2 — advisory key + log).

```

```sql

-- Domains: document-search.md (Reducto pipeline), document-ingestion.md (5 paths).
```

A reader (you, the freelancer, Sharad) should know in 5 seconds what each file does without opening 3 others. PRs without file-header summaries get blocked.

## File organisation

- Code follows the tree in `docs/ARCHITECTURE.md`. New top-level folders require updating that doc in the same PR.
- Every server action lives in `packages/lib/<domain>/`. Domain matches a file in `docs/domains/`.
- One public export per file (clean tree-shaking + readable imports).
- Tests live next to source as `*.test.ts`. Integration tests in `apps/web/tests/integration/{owner,client,worker}/`. RLS tests in `apps/web/tests/rls/`. E2E in `apps/web/tests/e2e/`.
- Per-tree ESLint rules (`packages/lib/eslint-config/{owner,client,worker}.js`) block cross-tree imports. Don't fight them — that boundary is the point.
- Component primitives in `packages/ui/`. Never inline colors / fonts / spacing — use `tokens.ts`.

## Code style

- Functional components + hooks + server actions. No classes unless wrapping a third-party SDK.
- Imports sorted: third-party → `@/packages/*` → `@/apps/*` → relative. Alphabetical within each group.
- Functions get a docstring: purpose, params, returns, throws, side effects.
- Variables: full words. Abbreviations allowed only for `id`, `db`, `ctx`, `req`, `res`.
- ASCII diagram in code comments for non-obvious data flow / state machines (per `DESIGN.md` and the eng-review canonical examples).

## Modular build — what "modular with integration tests" means

- **Each slice is a module.** Schema + RLS + server actions + UI + tests ship together. Slice N can't depend on slice N+1's table existing.
- **Cross-slice references go through `packages/lib/`.** UI of slice 08 calls `packages/lib/billing/invoice.action.ts`, never reaches into slice 09's directory.
- **Every slice has a paired integration test** in `apps/web/tests/integration/<role>/<slice>.test.ts` covering the slice's stated acceptance criteria. The integration test runs the server action end-to-end against a seeded Supabase branch DB. If acceptance criterion #N is checked off, integration test asserts #N.
- **RLS test pattern** (per `domains/rls.md`): cross-tenant SELECT returns empty, cross-tenant UPDATE fails, role-restricted operations enforce. Paired with every migration.
- **CI gate**: lint + typecheck + unit + integration + RLS + migration grep + schema-dump-diff + e2e stub. All required for merge.

## What you must never do

- Hardcode `stable_id` or `user_id`.
- Skip an RLS policy.
- Write a migration without a paired RLS policy + paired RLS test in the same PR.
- Call the LLM with raw user input. Wrap PDF text in `<document>` tags, use the system prompt per `domains/ai-extraction.md`.
- Use `localStorage` / `sessionStorage` / `IndexedDB` for sensitive data. Worker offline tap queue (slice 13) uses IndexedDB only for `{task_id, completed_at, version}` — no PII.
- Commit secrets. Use Vercel env vars (`.env.local` for local dev only, gitignored).
- Add a dependency without confirming: actively maintained, EU/Swiss data-residency compatible, **pinned to exact version (no `^`)** for `swissqrbill`, `@react-pdf/renderer`, `rrule.js`, `decimal.js`.
- Push directly to `main`. Branch + PR + 1 review.
- Mark a task done without manually clicking through it in the browser (see Workflow #3).
- Skip the file-header summary.

## What you should always do

- Start a non-trivial task in plan mode. State the file-tree change + test strategy before writing code.
- Run `pnpm lint && pnpm typecheck && pnpm test` before pushing.
- Write the test first when fixing a bug. Reproduce in a failing test, then fix.
- When a server action mutates, return the updated entity in the `ok: true, data` branch (not just a status flag).
- Use `packages/lib/auth/tenant-scope.ts` when querying across join boundaries (people ↔ horses ↔ events ↔ documents).
- Update the slice spec's acceptance-criteria checkboxes as you complete each one.
- For UI changes, manually verify against the slice's mockup in `docs/mockups/` AND test golden path + 1 edge case in the browser.
- Add a `TODO` comment ONLY if there's a paired entry in `docs/TODOS.md`. Floating TODOs in code without doc reference get flagged in review.

## Debugging — where to look when something breaks

When you (or the freelancer) need to diagnose an error:

1. **Sentry** — error capture with `stable_id` + `user_id` + `slice` + `action` + `error_class` tags. Filter by tag. Frankfurt EU instance per `domains/security-headers.md`.
2. **Better Stack / Axiom** — Pino structured logs, 30-day retention, EU region. Search by `stable_id` or `request_id`.
3. **Supabase dashboard** — query logs, slow queries, RLS rejection counts, pgvector index health.
4. **`/owner/inbox/documents` + `/owner/billing/invoices/sync-issues`** — owner-visible failures (Reducto pipeline + Bexio sync). Many "support tickets" actually have the answer here.
5. **`runbooks/oncall.md`** — incident playbooks per failure class (PDF gen, Bexio sync, magic-link bounce, DB pool, cron miss).
6. **`domains/*.md` failure-mode tables** — every named rescue path (`SwissQrSchemaError`, `BexioRefreshTokenRace`, `ReductoApiUnreachable`, etc.) has trigger / UX / recovery rows. Grep the error class.

Every server action logs entry + exit at INFO with arguments redacted of PII. Every rescue path logs at ERROR with the named error class. If you can't find a failure in the logs, the action probably didn't log on entry — fix the action, not the user's complaint.

## Anti-patterns to flag in review

- Direct Supabase client calls inside React components → move to a server action
- A query without `stable_id` filter → fix immediately
- A `try/catch` that swallows the error → log + return `{ ok: false, error: TypedError }`
- A hardcoded language string → move to `packages/i18n/`
- A floating-point amount → convert to integer cents
- A migration without RLS → block the PR
- A file without a header summary → block the PR
- A `TODO` without a `TODOS.md` entry → either fix it or document why deferred
- Inline color / font / spacing → use `packages/ui/tokens.ts`
- New abstraction with one caller → inline it
- Using `any` to bypass a type check → never; either explain why or fix the type
- A server action that throws instead of returning the discriminated union → rewrite
- Cross-tree import (worker importing from `packages/lib/billing/`) → ESLint should catch; if you bypass it, you're wrong
- A `cacheTag('something')` without the `stable:${id}:` prefix → cross-tenant leak (see `domains/cache-strategy.md`)
