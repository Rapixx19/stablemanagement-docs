# Domain: Migrations

How we change the schema safely. Every migration ships with a paired RLS policy and test.

## Numbering

Sequential, zero-padded, three-digit. The number is the merge order, not creation order:

```
supabase/migrations/
├── 001_init.sql
├── 002_people_horses.sql
├── 003_stalls.sql
├── 004_schedule.sql
├── 005_tasks.sql
├── 006_services.sql
├── 007_subscriptions_tabs.sql
├── 008_invoices.sql
├── 009_bexio.sql
├── 010_comms.sql
├── 011_requests_slots.sql
├── 012_worker_consent.sql
├── 013_timeclock.sql
└── 014_shifts_unavail.sql
```

If two devs branch from `006`, the second to merge re-numbers to `008`.

## Anatomy of a migration

Each file: schema change (CREATE / ALTER / DROP), indexes, RLS policies, seed data (if reference data).

Example skeleton:

```sql
-- 002_people_horses.sql — Slice 01
create table people ( ... );
create table horses ( ... );

create index ...;

alter table people enable row level security;
create policy people_isolation on people for all
  using (stable_id in (select auth.current_stable_ids()));
-- repeat policies for horses, horse_documents
```

The paired RLS test lives at `apps/owner/tests/rls/people.rls.test.ts`.

## The unbreakable rule

**Never edit a merged migration.** If you need to change something, add a new migration.

Reason: production has run that exact SQL. Editing it locally diverges history. Always forward.

## Reversibility

V1 is forward-only — no `down` migrations. To undo: write a new migration that reverses the change, test in preview, apply in prod. Most schema changes can't be cleanly reversed (data drops); explicit forward fixes are safer.

## Destructive changes

DROP COLUMN, DROP TABLE: requires a 2-phase rollout.

1. **Phase 1**: deploy code that no longer reads from the column. Don't drop yet.
2. **Phase 2**: after a deploy cycle (≥ 24h), drop the column.

This avoids the "deploy is mid-flight, half the servers reference the column you dropped" race.

## Adding a column with a default

For a non-trivial table, an ALTER with default can lock the table. On Postgres 11+, default-with-volatile-expression is fast (metadata-only). For tables with > 100k rows, do:

```sql
alter table tab_lines add column new_col text;
update tab_lines set new_col = 'x' where new_col is null; -- in batches if huge
alter table tab_lines alter column new_col set not null;
```

V1's largest table (`tab_lines`) won't exceed 100k rows for years.

## Indexes — concurrent

Build indexes with `concurrently` to avoid lock:

```sql
create index concurrently if not exists ix_tab_lines_person_period
  on tab_lines (stable_id, person_id, occurred_on);
```

`concurrently` cannot run inside a transaction. Mark migrations using it appropriately (Supabase CLI has a flag).

## RLS pairing rule

Every migration that creates a new table:

1. Creates the table
2. `alter table X enable row level security;`
3. Creates at least one policy
4. The corresponding RLS test in `apps/*/tests/rls/` is committed in the same PR

CI grep step: any migration creating a multi-tenant table (matches `stable_id`) without `enable row level security` in the same file → fail.

## Seed data

Reference tables (`vat_rates`, etc.) seeded inside the migration:

```sql
insert into vat_rates (code, rate, label, effective_from)
values ('STD', 0.081, 'Normalsatz', '2024-01-01')
on conflict (code) do nothing;
```

`on conflict` makes seeds idempotent.

## Test data

Never seed test data via a migration. Use `packages/db/seed.ts` for dev/test seeds. Seeds run via `pnpm db:seed`.

## Running migrations

- **Dev**: `supabase db push` against local Supabase
- **Preview**: applied automatically on Vercel preview deploy via Supabase branch
- **Prod**: applied via `supabase db push --linked` from CI on merge to `main`

CI fails if a migration errors. Roll forward, never roll back.

## Schema dump as source of truth

Canonical: `packages/db/schema.sql`. Generated by `supabase db dump` after migration apply. Reviewed in PRs to spot unintended drift.

## Hard rules

- One migration per merged PR, ideally
- Numbered files, never edit-after-merge
- Every multi-tenant table: RLS + paired test in the same PR
- `concurrently` for indexes on big tables
- Two-phase rollout for destructive changes
- Schema dump committed as source of truth

## Common pitfalls

- Renumbering an already-merged migration → add a new one instead
- Editing a merged migration locally → schema drift, painful to fix
- ALTER TABLE without thinking about lock duration → outage
- Dropping a column the deploy still reads → 500s
- Forgetting the paired RLS test → tenant leak ships undetected
