# Domain: i18n — DE / FR / IT

Switzerland is officially quadrilingual (DE, FR, IT, RM). This product ships with three: DE, FR, IT. Romansh speakers also speak German.

## Why all three from day one

DACH-only competitors don't translate. France-built tools don't ship CHF. Adding IT later means re-templating every email, invoice, and PDF — better to do it right once. Our research showed Bern + Vaud + Fribourg + Jura + Neuchâtel + Geneva hold ~49% of Swiss horses; French is disproportionately important. Ticino is small but symbolically critical.

## Library

`next-intl` for the Next.js apps. ICU MessageFormat for plurals, gender, dates, numbers. Translation files in `packages/i18n/messages/`:

```
packages/i18n/
├── messages/
│   ├── de.json
│   ├── fr.json
│   └── it.json
├── format.ts          // date/number/currency formatters
└── locale-detect.ts   // user → tenant → fallback
```

## File structure

Nested by feature/domain:

```json
{
  "common": {
    "save": "Speichern",
    "cancel": "Abbrechen"
  },
  "horses": {
    "list": {
      "title": "Pferde",
      "empty": "Noch keine Pferde",
      "addCta": "Pferd hinzufügen"
    }
  },
  "invoices": {
    "status": {
      "draft": "Entwurf",
      "sent": "Gesendet",
      "paid": "Bezahlt",
      "overdue": "Überfällig"
    }
  }
}
```

## Locale resolution

Per-message: `locale = user.preferred_locale ?? stable.default_locale ?? 'de'`. For broadcasts: respect each recipient's language.

For PDF generation: invoice always in the *recipient's* language (client preference), regardless of who triggered the action.

## Number formatting

Switzerland-correct:

- DE: `1'820.50` (apostrophe thousands, dot decimal)
- FR: `1 820,50` (space thousands, comma decimal)
- IT: `1'820.50` (Ticino style; matches DE not Italy IT-IT)

Use `Intl.NumberFormat('de-CH')` etc. Wrappers in `packages/i18n/format.ts`:

```ts
formatChf(182050, 'de') → "CHF 1'820.50"
formatChf(182050, 'fr') → "1 820,50 CHF"
formatChf(182050, 'it') → "CHF 1'820.50"
```

## Date formatting

Swiss style: `DD.MM.YYYY`. Use the locale's variation:

- DE: `04.05.2026`
- FR: `04.05.2026`
- IT: `04.05.2026`

Long-form differs per locale (`Montag, 4. Mai 2026` / `lundi 4 mai 2026` / `lunedì 4 maggio 2026`).

## VAT label

Mandatory per `domains/vat.md`:

- DE: `MWST`
- FR: `TVA`
- IT: `IVA`

Never `VAT`.

## Translation workflow

1. Developer adds key in `de.json` (source-of-truth)
2. Run `pnpm i18n:check` — fails if FR/IT missing the key
3. CI blocks merge if mismatched
4. Translation pass before pilot: native speaker reviews each language
5. Email templates and PDF templates have their own pass (legal language)

## LLM translation as draft, not final

Claude/DeepL can produce draft FR/IT from DE source. **Always reviewed by native speaker before ship.** The pilot phase relies on Ferdinand's family network for Ticino review; a paid translator service is in the V1.1 budget for ongoing churn.

## Strings that must not be translated

- Stable names, horse names, person names (proper nouns)
- UELN, microchip, FEI ID (standardized identifiers)
- UID-MWST format (only the suffix changes)
- Currency code `CHF`
- File extensions

## Right-to-left support

Not needed in V1. If we ship Arabic / Hebrew later (unlikely), the codebase needs LTR-RTL audit.

## Pluralization

ICU plurals handle DE/FR/IT correctly:

```json
"horses_count": "{count, plural, =0 {Keine Pferde} one {1 Pferd} other {# Pferde}}"
```

Gender forms in FR are explicit, not interpolated:

```json
"client_singular_m": "le client",
"client_singular_f": "la cliente"
```

For now, default to masculine for mixed/unknown — Swiss French follows that convention.

## Hard rules

- No hardcoded user-facing strings. Every visible string is a key.
- Keys are descriptive, not the source string. `horses.list.empty` not `noHorsesYet`.
- Reuse `common.*` for verbs (save, cancel, delete, edit, view).
- Date / number / currency always through wrappers, never raw `toFixed()`.
- Server-rendered pages set `<html lang="...">` per request locale.

## Testing

Per slice acceptance: at least one render in each of DE/FR/IT verifies layout doesn't break (German strings are typically 30% longer than English).

## Out of V1

- English (we serve Switzerland; English-speaking clients use one of DE/FR/IT)
